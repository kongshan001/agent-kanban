# v0.1 MVP 回顾与问题记录

## 日期: 2026-05-02

## 验证场景
在 agent-kanban 项目本身上跑一个真实任务：添加 GitHub Issue/PR 模板。

## 发现的问题

### P0 - 阻塞性问题

#### 1. 评估器是空壳，无法真正评审
- **现象**: `kanban task evaluate` 只是生成空模板文件，评分永远是 1.0
- **影响**: 自迭代循环无法真正工作，会一直空转到第 5 轮然后要求用户介入
- **根因**: evaluator.py 没有接入 LLM，只是创建空的 markdown 文件
- **v0.2 方案**: 接入 LLM 评估，支持配置评估 prompt 和 model

#### 2. PhaseGuard 检查逻辑过于僵化
- **现象**: 任务"添加 GitHub 模板"不需要写测试代码，但 PhaseGuard 强制要求测试文件
- **影响**: 用户被迫为不需要测试的任务编造测试
- **根因**: `check_execute_complete()` 硬编码了 `has_test_files()` 检查
- **v0.2 方案**: 支持跳过特定检查（在任务配置或 config.yaml 中设置 `skip_checks: ["test"]`）

### P1 - 体验问题

#### 3. 主分支名默认 main 但很多项目用 master
- **现象**: init 后 worktree 创建失败（已在测试中修复）
- **状态**: ✅ 已修复（自动检测当前分支）

#### 4. venv 被提交到 git
- **现象**: `pip install -e .` 创建的 venv 被 git 跟踪
- **状态**: ✅ 已修复（添加 .gitignore）

#### 5. agent run 只是打印 prompt，不实际执行
- **现象**: `kanban agent run TK-001` 只输出要执行的 prompt，不调用 claude/opencode
- **影响**: 无法端到端自动化
- **v0.2 方案**: 实现真正的 agent 调度（claude -p / opencode run）

### P2 - 架构问题（参考"Sub-Agent VS Agent Team"文章）

#### 6. 评估阶段的 4 角色拆分是"按角色拆"而非"按上下文边界拆"
- **现象**: tech_lead/qa/product/design 看的是同一份代码，上下文完全重叠
- **影响**: 4 次 LLM 调用做本可以 1 次完成的事，浪费 token 且 handoff 丢信息
- **v0.2 方案**: 改为单 Agent 综合评估，输出一份统一报告

#### 7. 线性流程不支持并行子任务
- **现象**: Plan→Execute→Evaluate 是严格串行
- **影响**: 多个独立子任务无法并行执行
- **v0.2 方案**: 支持 parallel 原语，允许任务内定义并行子任务
